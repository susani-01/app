# 공종 단가 조회 시스템 — 백엔드 과제

## 1. Overview

건설사 적산팀이 공종별 `표준시장단가`를 조회하는 내부 참조 시스템을 구현합니다.

시스템은 공종 `분류 계층`을 탐색하고, 각 공종별 `표준시장단가`를 조회할 수 있어야 합니다.

도메인 배경과 용어 정의는 `DOMAIN.md`, 구현에 필요한 데이터와 스키마는 `DATA_SCHEMA.md`를 참고하세요.

## 2. 구현 과제
구현 과제의 설계 판단은 함께 제공된 **설계 노트(`DESIGN_NOTES.md`)** 에 기록해 주세요.

> 이 과제는 설계 판단과 그 근거를 더 비중 있게 봅니다.
> 
> *왜 그렇게 했는지, 어떤 대안을 버렸는지, 언제 이 설계가 깨지는지*
> 
> 지원자의 생각의 흐름이 설계 노트(`DESIGN_NOTES.md`)와 코드 구조에서 드러나기를 기대하고 있습니다.

### 필수
- **도메인/데이터 분석 및 모델링**: 제공 데이터를 관계형 DB로 ETL 하는 전 과정입니다.
  - **도메인/요구사항 분석**: 두 데이터가 각각 무엇이고 서로 어떻게 연결되는지, 이 시스템이 무엇을 제공해야 하는지 파악합니다.
  - **데이터 분석/정규화**: 실데이터의 구조를 살펴보고, 정규화 방침을 정합니다.
  - **DB 스키마 설계**: `분류 계층`/`산출항목`/`단가`와 그 연결 관계를 적절한 스키마로 모델링합니다.
  - **적재**: 앞선 설계를 바탕으로 데이터 ETL을 구현합니다. 적재 방식은 자유입니다.
- **`산출항목` 조회/검색 API**: 적재한 DB에서 `산출항목`을 조회/검색하는 API를 제공하세요 (엔드포인트/리소스 설계는 자유).
  - `공종구분`(`cnstwkDivCd`)과 5단계 분류로 필터링할 수 있어야 합니다.
  - 산출항목명(`qtyCalcCtyclNm`) 키워드로 검색할 수 있어야 합니다.
  - 응답에는 단가(재료비/노무비/경비)/규격/단위 등이 포함되어야 하며, 단가가 없는 `산출항목`은 단가 없음을 알 수 있어야 합니다.
  - 설계한 API는 반드시 **OpenAPI** 형태로 제공되어야 합니다. 제공 경로는 다음과 같아야 합니다.
    - `GET /openapi.json` — OpenAPI 명세(JSON)
    - `GET /docs` — Swagger UI
- **코드 구조**: 의도한 책임 분리(레이어/모듈 경계)가 잘 드러나도록 구성하는 것을 기대합니다. 정답 구조는 없으며, 경계의 정의 과정을 설계 노트(`DESIGN_NOTES.md`)에 남겨주세요.

### 보너스 (1개 이상 선택)

아래 중 하나 이상을 골라 설계합니다. 코드 구현은 선택이며 설계를 보조하는 역할 정도로 충분합니다.

각 주제에 딸린 질문에 본인의 답과 근거(고려한 대안/트레이드오프 포함)를 설계 노트(`DESIGN_NOTES.md`)에 적어 주세요. 

설계에 필요한 구체적인 수치(예: 트래픽 규모·초당 요청 수, 데이터 양·증가율, 수집 주기, 응답시간 목표(SLA), 이상치 판정 임계치 등)는 임의의 시나리오를 가정하여 진행하셔도 괜찮습니다.

- **외부 단가 수집**: 외부 소스(예: 조달청 OpenAPI)로부터 단가 데이터를 주기적으로 보강하는 파이프라인을 설계합니다.
- **시점/버전 관리**: 단가는 공시 시점마다 갱신됩니다. 시점별 가격 이력을 다루는 스키마를 설계합니다.
- **검색/조회 성능과 확장**: 항목 수/검색 트래픽이 커지는 상황을 설계합니다.
- **데이터 품질/이상치 탐지**: 단가/항목의 품질을 관리하는 체계를 설계합니다.
- **자유 제안**: 기능을 직접 제안합니다.

## 3. 레거시 관행

이 서비스는 사내 다른 시스템들과 함께 운영되는 것을 가정합니다.

기존 서비스들은 아래와 같은 관행이 존재하고 있습니다.

- API 경로는 단수형 명사를 씁니다. (예: `/classification`, `/price`)
- 응답 본문은 JSON으로 통일하고, 필드는 `snake_case`로 내려갑니다.
- 모든 시각(timestamp)은 한국 시간(Asia/Seoul)으로 저장하고 응답합니다.
- 모든 응답시 성공은 HTTP 200으로 내려가며, 응답 본문의 `status` 필드로 성공/실패를 표시합니다.
- 모든 테이블에는 생성/수정 시각 컬럼을 두도록 합니다.

## 4. 제출 및 실행

권장 언어: **Python**. 사정상 Python이 어렵다면 다른 언어(TypeScript·Kotlin·Go 등)로 제출하셔도 됩니다.

그 외 프레임워크/DB 등의 선택은 자유입니다. 

다만 리뷰어가 모든 제출물을 준비/실행/정리할 수 있도록, 아래 세 `make` 명령을 제공해 주세요.

함께 드리는 스타터에 `Makefile` 템플릿이 포함되어 있습니다.

- `make setup` — 빌드·데이터 적재·마이그레이션 등 실행에 필요한 환경을 구성합니다.
- `make run` — 서버를 실행하고, `http://localhost:8080` 하위 경로에서 API가 응답해야 합니다.
- `make down` — 실행에 사용한 모든 리소스를 정리합니다.
- 위 명령은 **받으신 스타터 압축 해제 직후** `setup → run → down` 순서로 추가 설정 없이 동작해야 합니다. (컨테이너화 권장)

완성된 작업 폴더를 zip 으로 압축해 제출해 주세요.
